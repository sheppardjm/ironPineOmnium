The validation gate block has three gates in this exact order (order is critical — see research pitfall 1):

**Gate 1: Hidden start time (VAL-04)**
Check `startDate.endsWith("T00:00:01Z")`. If true, return HTTP 200 with `{ error: "hidden_start_time" }`. This MUST precede any epoch arithmetic because T00:00:01Z parses as midnight UTC, which would falsely trigger the start-time check.

**Gate 2: Distance (VAL-01/VAL-02)**
Determine `isDay1` from `localDateStr === DAY1_DATE`. Select the per-day minimum: `DAY1_MIN_DISTANCE_METERS` or `DAY2_MIN_DISTANCE_METERS`. If `distanceMeters < minDistanceMeters`, return HTTP 200 with:
```json
{
  "error": "distance_too_short",
  "actualDistanceKm": <Math.round(distanceMeters / 10) / 100>,
  "minDistanceKm": <156 or 153>
}
```

**Gate 3: Start time window — Day 1 only (VAL-03)**
Guarded by `if (isDay1)`. Parse `startDate` to epoch seconds: `Math.floor(new Date(startDate).getTime() / 1000)`. Compute cutoff: `GUN_EPOCH_SECONDS + START_WINDOW_SECONDS`. If `startEpoch > cutoffEpoch`, return HTTP 200 with:
```json
{
  "error": "start_too_late",
  "actualStartTime": "<startDate ISO string>",
  "cutoffTime": "<new Date(cutoffEpoch * 1000).toISOString()>"
}
```

CRITICAL: Every return statement MUST include the `multiValueHeaders` spread pattern: `...(Object.keys(multiValueHeaders).length ? { multiValueHeaders } : {})`. Dropping this causes token-refresh cookies to be lost on error paths (see research pitfall 4).

Do NOT add any validation to submit-result.js — the architecture validates at fetch time only (see research anti-pattern 6).
Do NOT use `start_date_local` for epoch comparison — only use `startDate` (which is `activity.start_date`, the true UTC ISO string).
Do NOT apply the start-time window check to Day 2 activities — VAL-03 is Day 1 only.
