**Part A — Sticky Rider column (all viewports):**

In `src/styles/global.css`, inside `@layer components`, add the following rules after the existing `.leaderboard-table tbody tr:hover` block and before `.leaderboard-table td strong`:

```css
.leaderboard-table th:nth-child(2),
.leaderboard-table td:nth-child(2) {
  position: sticky;
  left: 0;
  z-index: 2;
  background-color: var(--color-night-950);
}

.leaderboard-table thead th:nth-child(2) {
  z-index: 3;
}

.leaderboard-table tbody tr:first-child td:nth-child(2) {
  background-color: #0e1513;
}

.leaderboard-table tbody tr:hover td:nth-child(2) {
  background-color: rgba(15, 22, 20, 1);
}
```

Key details:
- `:nth-child(2)` targets the Rider column (2nd column after Rank/#).
- `background-color: var(--color-night-950)` makes the sticky cell opaque so scrolling columns don't bleed through.
- The first-row override (`#0e1513`) approximates the amber gradient overlay so the sticky cell doesn't look jarring against the highlighted row.
- The hover override ensures the sticky cell picks up a solid version of the hover background.
- `z-index: 3` on `thead th` ensures the sticky header cell sits above sticky body cells.

**Part B — Mobile breakpoint overrides:**

In `src/styles/global.css`, update the existing `@media (max-width: 639px)` block inside `@layer components`. Currently it contains:

```css
@media (max-width: 639px) {
  .cta-row > a,
  .cta-stack > a {
    width: 100%;
  }

  .leaderboard-table {
    min-width: 40rem;
  }
}
```

Replace the `.leaderboard-table { min-width: 40rem; }` rule with the following expanded mobile rules:

```css
.leaderboard-table {
  min-width: 0;
}

.leaderboard-table th,
.leaderboard-table td {
  padding: 0.75rem 0.45rem;
  font-size: 0.82rem;
  vertical-align: middle;
  white-space: nowrap;
}

.score-note {
  display: none;
}
```

Key details:
- `min-width: 0` removes the forced 40rem minimum that causes overflow at 375px. The table now sizes naturally and scrolls only when needed.
- Reduced padding and font-size keep columns compact.
- `vertical-align: middle` prevents misalignment when `.score-note` sub-rows are hidden.
- `white-space: nowrap` keeps score values on one line so columns don't wrap to multiple lines.
- `.score-note { display: none }` hides the raw duration/points sub-labels on mobile to save vertical space. These are secondary context; the numeric score is the primary data.

<task type="auto">
  <name>Task 2: Touch target safety — touch-action and spacing validation</name>
  <files>src/styles/global.css</files>
  <action>
In `src/styles/global.css`, inside `@layer components`, add `touch-action: manipulation` to the existing `.tab-button` rule block. Also add it to the `.search-input` rule (added by 09-01).

Find the existing `.tab-button` block and add `touch-action: manipulation`:

```css
.tab-button {
  min-height: 3rem;
  font-family: var(--font-mono);
  font-size: 0.82rem;
  letter-spacing: 0.06em;
  text-transform: uppercase;
  touch-action: manipulation;
}
```

Find the `.search-input` block (added by 09-01) and add `touch-action: manipulation`:

```css
.search-input {
  /* ... existing properties from 09-01 ... */
  touch-action: manipulation;
}
```

**What this does:** `touch-action: manipulation` eliminates the 300ms tap delay on mobile browsers and prevents accidental double-tap zoom on these interactive elements. This prevents the scenario where a rider taps a tab and it appears unresponsive (the 300ms delay) or where a double-tap on the search input triggers zoom instead of text selection.

**Why no row click handlers need attention:** The current `Leaderboard.astro` has no `onclick` handlers on `<tr>` elements — rows are purely display. This means SC-4 ("tapping a leaderboard row does not accidentally trigger search or tab navigation") is inherently satisfied by the absence of row interactivity. The only proactive measures needed are the touch-action rules on the controls that ARE interactive (tabs and search).

**Spacing validation:** The search input is placed in `.leaderboard-head` which uses `display: grid; gap: 1rem`. This 1rem (16px) gap between the search input and the tab list, and between the tab list and the first table row, provides sufficient spacing to prevent accidental taps on adjacent elements. No additional margin adjustments are needed.
