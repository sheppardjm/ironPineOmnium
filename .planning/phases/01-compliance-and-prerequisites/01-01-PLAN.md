  <action>
Create DATA-MODEL.md with the following sections:

**1. Per-Athlete JSON Schema** (`public/data/results/athletes/{athleteId}.json`)
Define the stored fields. Based on the existing `RiderResult` type in `src/lib/types.ts` and the phase context decisions:
- `athleteId`: string — Strava numeric athlete ID stored as string (not hashed; simplicity wins, and the ID alone does not constitute "Strava Data" under the ToS since it's a numeric identifier, not a profile field)
- `displayName`: string — rider-chosen, locked after first submission
- `category`: "men" | "women" | "non-binary" — self-reported, locked after first submission
- `day1`: object or null — `{ movingTimeSeconds: number, submittedAt: ISO8601 string, activityId: string }`
- `day2`: object or null — `{ sectorEfforts: Record<string, number> (segmentId -> elapsed seconds), komSegmentIds: string[] (segments where this rider had an effort), submittedAt: ISO8601 string, activityId: string }`
- `createdAt`: ISO8601 string
- `updatedAt`: ISO8601 string

Note: `activityId` is stored for operational deduplication (same activity resubmitted) but is NOT displayed anywhere public.

**2. Public Leaderboard Display Fields**
Define what the public leaderboard renders per row. Must contain ONLY computed values and rider-chosen identity:
- Rank (computed)
- Display name (rider-chosen)
- Category (rider-chosen)
- Day 1 Score (computed points, NOT raw moving time)
- Day 2 Sector Score (computed points, NOT raw sector times)
- KOM Score (computed points)
- Total Score (computed points)

Explicitly list what is NOT shown publicly: Strava profile name, avatar URL, Strava profile link, raw moving time, raw sector times, activity URLs, hometown (per context decision: not collected).

**3. Rider's Own Submission View (Session-Only)**
Define what the rider sees during their own submission confirmation (Phase 5 scope, but the boundary must be documented now):
- Their raw moving time mapped to points ("Your moving time: 4:12:33 -> 35.0 pts")
- Their raw sector times mapped to points per sector
- Their KOM points breakdown
This data is computed in the submission flow and shown in the browser session. It is NOT stored in the public JSON and NOT visible to other riders.

**4. Strava ToS Compliance Summary**
State the compliance boundary:
- No Strava profile fields (firstname, lastname, profile_medium, city) are stored or displayed
- No raw Strava activity data (moving_time, elapsed_time, segment effort times) is displayed publicly
- Computed point scores are derived values, not "Strava Data" under the API Agreement
- Rider identity on the leaderboard is self-reported (display name + category), not pulled from Strava
- The only Strava-originated data stored in public JSON files is: athleteId (numeric identifier for deduplication) and activityId (for operational deduplication, not displayed)

**5. Re-submission and Identity Rules**
- First submission: rider provides display name + category; these are locked
- Day 2 submission: system retrieves existing athlete file, reuses identity fields
- Re-submission for same day: new submission replaces old (overwrite day1 or day2 object in JSON)
- Strava deauth: athlete JSON file is deleted entirely
