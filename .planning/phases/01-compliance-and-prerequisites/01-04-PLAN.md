**Step 1: Create `scripts/sample-fallback.csv`**

The CSV template must match the per-athlete JSON schema from the approved data model (01-01). Columns:

```csv
display_name,category,day1_moving_time_seconds,day2_sector_SEGID1,day2_sector_SEGID2,...,day2_kom_SEGID1,day2_kom_SEGID2,...
"Jane Doe",women,15153,420,380,...,yes,no,...
```

Where:
- `display_name`: string — rider-chosen name
- `category`: "men" | "women" | "non-binary"
- `day1_moving_time_seconds`: integer — total moving time in seconds (can be empty if rider only did Day 2)
- `day2_sector_{segmentId}`: integer — elapsed seconds for each timed sector (use actual segment IDs from `src/lib/segments.ts` as column suffixes)
- `day2_kom_{segmentId}`: "yes" | "no" — whether the rider completed this KOM segment (use actual segment IDs)

Include 2-3 example rows showing: a rider who did both days, a rider who only did Day 1, a rider who only did Day 2.

**Step 2: Create `scripts/csv-fallback.ts`**

A local Node.js/TypeScript script (run with `npx tsx scripts/csv-fallback.ts path/to/results.csv`). The script:

1. Reads the CSV file from the command-line argument
2. Parses each row into a rider object
3. Validates:
   - `category` is one of the valid category IDs (import from `src/lib/types.ts`)
   - Day 2 sector column suffixes match `SECTOR_SEGMENT_IDS` from `src/lib/segments.ts`
   - Day 2 KOM column suffixes match `KOM_SEGMENT_IDS` from `src/lib/segments.ts`
   - At least one of day1 or day2 data is present per row
4. Assigns fallback athlete IDs: `manual-001`, `manual-002`, etc. (zero-padded, sequential)
5. Builds per-athlete JSON matching the schema from the data model:
   ```json
   {
     "athleteId": "manual-001",
     "displayName": "Jane Doe",
     "category": "women",
     "day1": { "movingTimeSeconds": 15153, "submittedAt": "...", "activityId": "manual" },
     "day2": { "sectorEfforts": { "SEGID1": 420, "SEGID2": 380 }, "komSegmentIds": ["SEGID1"], "submittedAt": "...", "activityId": "manual" },
     "createdAt": "...",
     "updatedAt": "..."
   }
   ```
6. For each athlete JSON, commits to GitHub via the Contents API:
   - GET `https://api.github.com/repos/{owner}/{repo}/contents/public/data/results/athletes/{athleteId}.json` to check if file exists and get SHA
   - PUT the file with base64-encoded content, commit message, and SHA if updating
7. Uses environment variables: `GITHUB_TOKEN`, `GITHUB_OWNER`, `GITHUB_REPO` (read from `.env` or process.env)
8. Logs each commit result (success/failure) to stdout
9. Does a dry-run by default — requires `--commit` flag to actually write to GitHub

Dependencies: Use only Node.js built-in modules plus `csv-parse` (or manual CSV parsing if you want zero dependencies). Do NOT add heavy frameworks. If using `csv-parse`, note it in the script header so the user knows to install it.

Error handling: If any row fails validation, log the error and skip that row (don't abort the entire batch). Print a summary at the end: X succeeded, Y failed, Z skipped.

Create CSV-FALLBACK.md documenting the step-by-step procedure for event day fallback. This is an operational runbook for the event organizer. Structure:

**When to use this:** Strava athlete limit review was not approved by June 6, 2026, OR Strava API is unavailable on event day.

**Prerequisites:**
- GitHub personal access token with repo write access (stored in `.env` as `GITHUB_TOKEN`)
- `GITHUB_OWNER` and `GITHUB_REPO` set in `.env`
- Node.js 18+ and `npx tsx` available locally

**Data collection workflow:**
1. At the event finish line, collect from each rider: display name, category, Strava activity URL (or have them share the activity directly)
2. For Day 1: note the rider's moving time from the Strava activity page (visible publicly or via shared link)
3. For Day 2: note each rider's segment effort times for the timed sectors and KOM segments from the Strava activity page
4. Enter the data into the CSV template (`scripts/sample-fallback.csv` as reference)

**Script execution:**
1. Copy `scripts/sample-fallback.csv` to `scripts/event-results.csv`
2. Fill in rider data
3. Dry-run: `npx tsx scripts/csv-fallback.ts scripts/event-results.csv`
4. Review output for validation errors
5. Commit: `npx tsx scripts/csv-fallback.ts scripts/event-results.csv --commit`
6. Verify athlete JSON files appear in the GitHub repo
7. Trigger a Netlify rebuild (or wait for the automatic rebuild) to update the leaderboard

**Handling corrections:**
- To fix a rider's data: edit the CSV row and re-run with `--commit` (the script updates existing files by SHA)
- To remove a rider: manually delete their JSON file from the GitHub repo

**Limitations:**
- KOM ranking is not automatically computed across riders in fallback mode — the script records which KOM segments each rider completed, but cross-rider comparison must be done manually or via a separate scoring pass
- Display names assigned in fallback mode will need to be reconciled if/when the Strava integration becomes available
