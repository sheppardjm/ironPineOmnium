<task type="checkpoint:human-action" gate="blocking">
  <name>Task 1: Configure Netlify dashboard and link site</name>
  <action>
The following steps require access to the Netlify dashboard and cannot be automated by Claude:

**Step 1: Link the local project to your Netlify site**
Run in the project root:
```bash
npx netlify link
```
Select the existing Netlify site for ironPineOmnium (or create one if it doesn't exist yet). This creates a `.netlify/` directory locally with the site ID.

**Step 2: Create a Build Hook**
1. Go to Netlify Dashboard -> Site configuration -> Build & deploy -> Build hooks
2. Click "Add build hook"
3. Name it "submission-trigger"
4. Select the main branch
5. Copy the generated URL — you'll use it as `NETLIFY_BUILD_HOOK`

**Step 3: Set all 8 environment variables in Netlify Dashboard**
Go to: Netlify Dashboard -> Site configuration -> Environment variables -> Add a variable

| Variable | Value Source |
|----------|-------------|
| `STRAVA_CLIENT_ID` | Strava API Settings -> My API Application -> Client ID |
| `STRAVA_CLIENT_SECRET` | Strava API Settings -> My API Application -> Client Secret |
| `STRAVA_REDIRECT_URI` | `https://{your-site}.netlify.app/api/strava-callback` |
| `STRAVA_VERIFY_TOKEN` | Generate: `openssl rand -hex 32` |
| `GITHUB_TOKEN` | GitHub Settings -> Developer settings -> Personal access tokens -> repo scope |
| `GITHUB_OWNER` | Your GitHub username |
| `GITHUB_REPO` | Repository name (ironPineOmnium) |
| `NETLIFY_BUILD_HOOK` | The URL from Step 2 |

All variables should use the default scope (All scopes) so they're accessible in both builds and functions.

**Step 4: Verify variables are set**
Run:
```bash
npx netlify env:list
```
All 8 variables should appear (values are masked).
  </action>
  <resume-signal>Type "done" when all 8 env vars are set and netlify link is complete, or describe any issues.</resume-signal>

<task type="auto">
  <name>Task 2: Verify netlify dev and smoke-test function</name>
  <files></files>
  <action>
1. Start `netlify dev` in the background and wait for it to be ready:
   ```bash
   npx netlify dev &
   ```
   Wait for the "Server now ready" message (or equivalent), then:

2. Test the health endpoint and verify env var injection:
   ```bash
   curl -s http://localhost:8888/api/health
   ```
   Expected response: `{"ok":true,"env":{"STRAVA_CLIENT_ID":true,"STRAVA_CLIENT_SECRET":true,...}}` — all 8 env vars should be `true`.

3. Assert health endpoint passes and all env vars are present:
   ```bash
   HEALTH=$(curl -s http://localhost:8888/api/health)
   echo "$HEALTH" | grep -q '"ok":true' && echo "PASS: health endpoint" || echo "FAIL: health endpoint"
   echo "$HEALTH" | grep -q '"STRAVA_CLIENT_ID":true' && echo "PASS: env injection" || echo "FAIL: env injection"
   # Check that no env var is false (missing)
   echo "$HEALTH" | grep -q 'false' && echo "FAIL: some env vars missing" || echo "PASS: all env vars present"
   ```

4. Verify no v2 syntax exists anywhere in the functions directory:
   ```bash
   grep -r "export default" netlify/functions/ && echo "FAIL: v2 syntax found" || echo "PASS: no v2 syntax"
   ```

5. Test that Astro pages still work through the proxy:
   ```bash
   curl -s http://localhost:8888/ | grep -q "<html" && echo "PASS: Astro served" || echo "FAIL: Astro not served"
   ```

6. Stop the background `netlify dev` process.

If any test fails, diagnose the issue by checking:
- `netlify dev` output for errors
- `netlify.toml` redirect rule syntax
- Function file syntax (must be `export const handler`, not `export default` or `exports.handler`)
- Whether `netlify link` was completed (check for `.netlify/state.json`)
  </action>
  <verify>
- `curl http://localhost:8888/api/health` returns `{"ok":true,"env":{...}}` with HTTP 200 and all 8 env vars are `true`
- `curl http://localhost:8888/` returns Astro HTML
- `grep -r "export default" netlify/functions/` returns no matches
- `npx netlify env:list` shows all 8 variables
  </verify>
  <done>`netlify dev` starts cleanly, /api/health returns 200 with {ok:true} and all 8 env vars confirmed present via the response body, Astro pages are served through the proxy, and no v2 function syntax exists.</done>
</task>
